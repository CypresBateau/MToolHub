#### 模块名称：三级路由决策机制
- 功能描述：
  - 实现 `RouteDecisionMaker` 路由决策器，基于向量检索分数自动决定调用模式
  - 三级阈值设计：
    - **direct_call**（分数 ≥ 0.85）：高置信度，直接选择 top-1 资源执行，无需 Claude 二次判断
    - **claude_select**（分数 0.60 ~ 0.85）：中置信度，取 top-3 候选资源交由 Claude 根据上下文选择最优
    - **chat_only**（分数 < 0.60）：低置信度，不调用任何工具，进入纯对话模式由 Claude 直接回答
  - 阈值支持通过 .env 环境变量动态配置（`DIRECT_CALL_THRESHOLD`、`CLAUDE_SELECT_THRESHOLD`）
  - 图像文件特殊处理逻辑：当用户上传图像文件时，强制将图像分析模型（MAVL）加入候选资源列表，确保影像分析需求不被漏判
  - 返回结构化 `RoutingPlan` 对象，包含：mode（路由模式）、confidence（置信度等级）、selected_resources（候选资源列表及分数）、trace（决策链路追踪）

#### 模块名称：Claude API 客户端封装
- 功能描述：
  - 封装 Anthropic Claude API，提供四大核心能力方法：
  - **`extract_parameters()`**：从自然语言中提取结构化参数。将工具的参数定义转换为 Claude tool_use 格式，调用 Claude 让其从用户消息中识别并提取出所有参数值，返回结构化字典。支持复杂参数类型、自动类型转换、处理缺失值（默认填 0）
  - **`interpret_result()`**：将工具的 JSON 输出转换为自然语言解释。传入用户原始问题、工具名称和原始结果，Claude 生成上下文相关的解释，包括专业术语说明、风险等级提示、后续建议
  - **`chat()`**：纯对话模式，不调用任何工具。用于向量检索置信度低于 0.60 的场景，或用户询问一般医学知识、无需工具的咨询
  - **`select_and_execute()`**：Claude 选择模式。将 top-3 候选资源转换为 Claude tools 格式，Claude 根据用户消息语义判断选择最合适的工具并调用，适用于置信度在 0.60-0.85 之间的模糊场景
  - 所有方法均为异步实现，支持超时控制（默认 30 秒）和异常捕获
- 产出形式：
  - Git commit：feat: Claude API client with parameter extraction & result interpretation

#### 模块名称：三类资源执行器
- 功能描述：
  - 设计 `Executor` 抽象基类，采用策略模式（Strategy Pattern），定义统一的 `execute()` 接口，返回标准化结果格式（success / result / trace / disclaimer）
  - **ToolExecutor（工具执行器）**：负责 MDCalc、单位换算、评分工具的调用。流程为：① 参数提取（已有则直接使用，否则调 Claude extract_parameters）→ ② 调用 Gateway（POST `http://gateway:9000/tools/{name}/call`，传入 function_name 和 arguments）→ ③ 结果解释（调 Claude interpret_result 生成自然语言）。支持 Gateway 连接失败重试 3 次
  - **ModelExecutor（模型执行器）**：负责 MAVL 胸片分析等影像模型调用。流程为：① 验证 file_bytes 存在 → ② 调用 Gateway（POST `http://gateway:9000/tools/{name}/predict`，multipart/form-data 传图像和 top_k 参数）→ ③ Claude 解释疾病概率。输出格式为 predictions 列表（disease + probability）
  - **SkillExecutor（技能执行器）**：负责加载并执行 SKILL.md 技能。根据技能类型分四种处理方式：
    - `document_only`：仅加载 SKILL.md 注入 system prompt，调 Claude chat
    - `tool_reference`：加载 SKILL.md + references/ 目录下参考文档（限 8000 字符），合并注入 system prompt
    - `executable`：加载 SKILL.md + coworker.py 中的 TOOLS 函数列表，将 Python 函数转为 Claude tool_use 格式，支持工具调用
    - `complex_workflow`：暂返回提示信息，留待后续实现
- 产出形式：

#### 模块名称：编排器（Orchestrator）
- 功能描述：
  - 实现 `Orchestrator` 编排器作为执行层的统一入口，接收 `RoutingPlan` 并协调执行

#### 模块名称：对话接口与直接执行接口
- 功能描述：
  - **`POST /api/chat`（对话接口）**：支持 multipart/form-data 格式，接收 message（文本）、file（可选，如胸片图像）、conversation_id（可选，会话 ID）。执行流程：接收请求 → 路由决策 → 编排器执行 → 返回响应。响应包含 response（回复文本）、tools_used（使用的工具列表）、routing_info（路由模式、置信度、候选资源、追踪信息）、disclaimer（免责声明）
  - **`POST /api/execute`（直接执行接口）**：支持 JSON 格式，接收 resource_id（如 "tool-mdcalc:wells_score_dvt"）、arguments（结构化参数字典）、context（可选上下文）。绕过路由决策直接执行指定资源，适用于前端已确定要调用哪个工具的场景
  - 两个接口均包含完整的 Pydantic 参数验证、错误处理和标准化响应格式
  - 所有 I/O 操作均使用 async/await 异步执行，使用 httpx.AsyncClient 连接池复用连接
- 产出形式：

#### 模块名称：自动化数据导入系统
- 功能描述：
  - **`scripts/import_from_gateway.py`**：自动从 Gateway 的 `/tools` 接口发现并导入所有已注册资源。识别逻辑：`type=remote, input=json` → 工具类（进一步调 `/tools/{name}/info` 获取函数列表和参数定义）；`type=remote, input=image` → 模型类。解析转换后生成 `data/registry/tools.json`（1108 个工具）和 `data/registry/models.json`（1 个模型）
  - **`scripts/import_skills.py`**：自动扫描 `skills/` 目录导入技能。类型检测规则：仅 SKILL.md → document_only；含 references/ → tool_reference；含 coworker.py（有 TOOLS 列表）→ executable；含大型数据/多子目录 → complex_workflow。元数据提取：从 SKILL.md 首个 # 标题取名称，首段取描述，Keywords/关键词行取关键词，自动检测中英文。最终导入 88 个技能（document_only: 45, tool_reference: 28, executable: 12, complex_workflow: 3）
  - 两个脚本均支持命令行参数自定义 Gateway URL 和输出目录

### （3）本周最大问题
- 三类资源（工具/模型/技能）的调用方式差异巨大：工具是 JSON RPC 式调用需要精确参数、模型是 multipart 文件上传、技能是 prompt 注入式对话。如何在统一的 Executor 接口下优雅地抽象这三种模式，并在编排器中以一致的方式处理，是本周设计上花费时间最多的部分。最终采用策略模式解决，但 SkillExecutor 内部仍需进一步按四种子类型分支处理，代码复杂度较高。
